209 | ## Python Parser (AST-based)
210 | 
211 | ### Implementation Details
212 | 
213 | 1. **Parse AST**: `ast.parse(code)` with error handling
214 | 2. **Walk nodes**: Use `ast.walk()` to traverse all nodes
215 | 3. **Extract functions**:
216 |    - Regular: `ast.FunctionDef`
217 |    - Async: `ast.AsyncFunctionDef`
218 |    - Capture: name, params with annotations, return annotation, decorators, docstring, line range
219 | 4. **Extract classes**:
220 |    - Node: `ast.ClassDef`
221 |    - Capture: name, base classes, decorators, docstring, line range
222 | 5. **Extract imports**:
223 |    - `ast.Import`: module names
224 |    - `ast.ImportFrom`: module + imported names
225 | 6. **Extract top-level variables**:
226 |    - `ast.Assign` at module level (col_offset == 0)
227 |    - Skip if inside function/class scope
228 | 7. **Body summary**:
229 |    - Walk function body statements
230 |    - Map statement types to human-readable strings
231 |    - Truncate values, max 8 lines
232 | 8. **Function calls**:
233 |    - Find `ast.Call` nodes in function body
234 |    - Extract function name from `ast.Name.id` or `ast.Attribute.attr`
235 |    - Deduplicate while preserving order
236 | 
237 | ### Body Summary Mapping
238 | ```python
239 | ast.Assign → "varname = value"
240 | ast.Return → "return value"
241 | ast.If → "if/else branch"
242 | ast.For → "for loop"
243 | ast.While → "while loop"
244 | ast.With → "with context manager"
245 | ast.Try → "try/except block"
246 | ast.Raise → "raise exception"
247 | ```

249 | ## Generic Parser (Regex-based)
250 | 
251 | ### Implementation Strategy
252 | 
253 | 1. **Import extraction**: Language-specific regex patterns
254 | 2. **Class extraction**: Match class/struct/interface definitions
255 | 3. **Function extraction**: Multiple patterns per language (regular, arrow, async, methods)
256 | 4. **Variable extraction**: Top-level only (skip indented lines)
257 | 5. **Body extraction**: Find function start, use block-end detection
258 | 6. **Block-end detection**:
259 |    - Track brace depth `{}`
260 |    - Handle string literals (skip braces in strings)
261 |    - Return line where depth returns to 0
262 |    - Ruby: track keyword depth (def/end, class/end, etc.)
263 | 7. **Body summary**: Regex patterns for assignments, returns, control flow
264 | 8. **Function calls**: `(\w+)\s*\(` with keyword filtering
265 | 9. **Return types**: Language-specific patterns (`: Type`, `-> Type`)
266 | 10. **Decorators**: Check line above function for `@`
267 | 11. **Parent class**: Check if function line falls within class range
268 | 
269 | ### Keyword Skip List for Call Extraction
270 | ```
271 | if, for, while, switch, case, return, new, throw, catch, 
272 | typeof, instanceof, sizeof, print, println, printf, fmt, 
273 | log, console, var, let, const, def, fn, func, function, 
274 | class, struct, enum, interface, type, impl, import, from, 
275 | require, include, use, package, pub, public, private, 
276 | protected, static, final, async, await, yield, match, where
277 | ```
278 | 
279 | ### Block End Detection Algorithm
280 | ```python
281 | def find_block_end(lines, start_line):
282 |     depth = 0
283 |     in_string = False
284 |     string_char = None
285 |     
286 |     for i, line in enumerate(lines[start_line:], start=start_line):
287 |         for char in line:
288 |             # Handle string literals
289 |             if char in ['"', "'", '`'] and not in_string:
290 |                 in_string = True
291 |                 string_char = char
292 |             elif char == string_char and in_string:
293 |                 in_string = False
294 |                 string_char = None
295 |             
296 |             # Track brace depth outside strings
297 |             if not in_string:
298 |                 if char == '{':
299 |                     depth += 1
300 |                 elif char == '}':
301 |                     depth -= 1
302 |                     if depth == 0:
303 |                         return i
304 |     
305 |     return start_line  # Fallback
306 | ```

const NODE_W = 300;
const NODE_H_BASE = 120;
const H_GAP = 150;
const V_GAP = 80;
const GROUP_PAD = 24;
const METHOD_H_GAP = 200;

// Estimate node height based on content
const estimateNodeHeight = (node) => {
  let height = 60; // Base height
  
  if (node.params && node.params.length > 0) {
    height += 40;
  }
  
  if (node.return_type) {
    height += 20;
  }
  
  if (node.body_summary && node.body_summary.length > 0) {
    height += 24 + (node.body_summary.length * 22);
  }
  
  if (node.docstring || node.ai_description) {
    height += 30;
  }
  
  height += 20; // Footer
  
  return Math.max(height, NODE_H_BASE);
};

// Build parent map for methods
const buildParentMap = (nodes) => {
  const parentMap = {};
  nodes.forEach(node => {
    if (node.parent) {
      parentMap[node.id] = node.parent;
    }
  });
  return parentMap;
};

// Find top-level owner of a node (class or standalone function)
const findTopLevelOwner = (nodeId, parentMap, nodes) => {
  const nodeMap = {};
  nodes.forEach(n => nodeMap[n.id] = n);
  
  let current = nodeId;
  while (parentMap[current]) {
    current = parentMap[current];
  }
  
  return current;
};

// Build call graph between top-level callables
const buildCallGraph = (nodes, edges) => {
  const parentMap = buildParentMap(nodes);
  const callGraph = {};
  const inDegree = {};
  
  // Initialize
  nodes.forEach(node => {
    if (node.type === 'function' || node.type === 'class') {
      const owner = findTopLevelOwner(node.id, parentMap, nodes);
      if (!callGraph[owner]) {
        callGraph[owner] = new Set();
        inDegree[owner] = 0;
      }
    }
  });
  
  // Build edges between top-level owners
  edges.forEach(edge => {
    const sourceOwner = findTopLevelOwner(edge.source, parentMap, nodes);
    const targetOwner = findTopLevelOwner(edge.target, parentMap, nodes);
    
    if (sourceOwner !== targetOwner && callGraph[sourceOwner] && callGraph[targetOwner]) {
      if (!callGraph[sourceOwner].has(targetOwner)) {
        callGraph[sourceOwner].add(targetOwner);
        inDegree[targetOwner]++;
      }
    }
  });
  
  return { callGraph, inDegree };
};

// BFS to assign depth levels
const assignDepthLevels = (callGraph, inDegree) => {
  const depths = {};
  const queue = [];
  
  // Find root nodes (no incoming edges)
  Object.keys(inDegree).forEach(nodeId => {
    if (inDegree[nodeId] === 0) {
      queue.push(nodeId);
      depths[nodeId] = 0;
    }
  });
  
  // BFS
  while (queue.length > 0) {
    const current = queue.shift();
    const currentDepth = depths[current];
    
    if (callGraph[current]) {
      callGraph[current].forEach(neighbor => {
        if (depths[neighbor] === undefined) {
          depths[neighbor] = currentDepth + 1;
          queue.push(neighbor);
        }
      });
    }
  }
  
  // Assign depth 0 to any remaining nodes
  Object.keys(callGraph).forEach(nodeId => {
    if (depths[nodeId] === undefined) {
      depths[nodeId] = 0;
    }
  });
  
  return depths;
};

## Usage

### Basic Analysis

1. Paste code into the editor (left panel)
2. Select language or use auto-detect
3. Click "Analyze Code"
4. Interact with the graph:
   - Click nodes to lock highlight on connections
   - Hover edges to temporarily highlight
   - Scroll to zoom, drag to pan
   - Click empty space to unlock

### Impact Analysis

1. Enable "Impact Mode" checkbox
2. Paste original code in top panel
3. Paste updated code in bottom panel (or use "Copy from above")
4. Click "Compare Impact"
5. View color-coded changes:
   - **Green**: Added functions
   - **Orange**: Modified functions
   - **Red**: Removed functions
   - **Purple**: Affected (calls modified functions)
   - **Gray**: Unchanged

## Color Scheme

### Node Types
- **Function**: Purple (#6C63FF)
- **Class**: Green (#4CAF50)
- **Variable**: Orange (#FF7043)
- **Import**: Cyan (#26C6DA)

### Edge Types
- **Calls**: Purple (#A59BFF)
- **Inherits**: Green (#81C784)
- **Imports**: Cyan (#80DEEA)

### Impact Status
- **Added**: Green (#4CAF50)
- **Modified**: Orange (#FF9800)
- **Removed**: Red (#f44336)
- **Affected**: Purple (#9C27B0)
- **Unchanged**: Gray (#555)

## Features in Detail

### Interactive Highlighting

- **Click a node**: Locks highlight on that node and all connected edges/nodes
- **Click same node again**: Unlocks and resets
- **Click empty space**: Unlocks and resets
- **Hover an edge** (when unlocked): Temporarily highlights that edge and its nodes
- **Hover an edge** (when locked): Ignored - lock takes priority

### Resizable Panel

- Drag the 6px divider between code editor and graph
- Minimum width: 250px
- Maximum width: window width - 400px
- Divider highlights purple on hover

### Tree Layout Engine

- Imports at top row
- Variables at second row
- Functions and classes organized by call depth (BFS)
- Methods grouped inside their parent class
- Class groups have dashed green borders
- Automatic spacing and centering

### Node Content

Each node displays:
- Type badge (FUNCTION, CLASS, METHOD, etc.)
- Name
- Impact badge (if in impact mode)
- Decorators (if any)
- Parameters
- Return type
- Logic summary (body statements)
- AI description (with "AI:" prefix on tinted background)
- Docstring (only if no AI description)
- Line range footer

## API Integration

The frontend communicates with the backend via three endpoints:

- `POST /parse` - Parse code and return graph data
- `POST /impact` - Compare two code versions
- `GET /health` - Check API status and supported languages

See `src/services/api.js` for implementation details.

### Graph not displaying
- Check that backend is running on port 8000
- Check browser console for errors
- Verify code is valid for the selected language

### Slow performance with large graphs
- Try reducing code size
- Disable minimap if needed
- Use zoom controls to focus on specific areas

### Layout issues
- Refresh the page
- Try re-analyzing the code
- Check that all nodes have valid data

## 🔌 API Endpoints

### POST /parse
Parse source code and return graph data.

**Request:**
```json
{
  "code": "def hello():\n    print('world')",
  "language": "python"  // optional, auto-detected if null
}
```

**Response:**
```json
{
  "nodes": [...],
  "edges": [...],
  "language": "python",
  "stats": {
    "functions": 5,
    "classes": 2,
    "variables": 3,
    "imports": 4,
    "connections": 8
  }
}
```

Analyze impact between two code versions.

**Request:**
```json
{
  "old_code": "...",
  "new_code": "...",
  "language": "python"  // optional
}
```

**Response:**
Same as `/parse` but nodes include `impact_status` field.

}

## 🎯 Key Features Explained

### Interactive Highlighting System
- **Click a node**: Locks highlight on that node and all connected edges/nodes
- **Click same node again**: Unlocks and resets to default state
- **Click empty space**: Unlocks and resets to default state
- **Hover an edge** (when unlocked): Temporarily highlights that edge and its nodes
- **Hover an edge** (when locked): Ignored - lock takes priority

### Tree Layout Engine
- Imports placed at top row
- Variables at second row
- Functions and classes organized by call depth using BFS
- Methods grouped inside their parent class with dashed borders
- Automatic spacing and centering for optimal visualization

### Impact Analysis Algorithm
1. Parse both code versions
2. Build name maps for functions
3. First pass: Identify added, modified, removed, unchanged
4. Second pass: Mark affected functions (call modified/added functions)
5. Display with color-coded badges
